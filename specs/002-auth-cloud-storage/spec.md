# Feature Specification: 인증 및 클라우드 저장소

**Feature Branch**: `002-auth-cloud-storage`
**Created**: 2026-05-22
**Status**: Draft
**Input**: 현재 Todo 앱에 실제 로그인 기능과 클라우드 저장소를 추가해주세요.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 이메일 회원가입 및 로그인 (Priority: P1)

신규 사용자가 이메일과 비밀번호로 계정을 만들고, 이후 같은 정보로 다시 로그인하여 자신의 할 일 목록에 접근할 수 있다.

**Why this priority**: 인증 시스템의 핵심이며, 이 스토리 없이는 나머지 기능이 의미가 없다. Google 로그인보다 먼저 구현되어야 기본 인증 흐름이 완성된다.

**Independent Test**: 이메일/비밀번호 회원가입 → 로그아웃 → 재로그인 → 할 일 목록이 유지되는지 확인으로 독립적으로 검증 가능.

**Acceptance Scenarios**:

1. **Given** 미가입 사용자가 로그인 화면에 있을 때, **When** 이메일과 비밀번호(8자 이상)를 입력하고 "회원가입" 버튼을 누르면, **Then** 계정이 생성되고 메인 화면으로 이동한다.
2. **Given** 이미 가입된 사용자가 로그인 화면에 있을 때, **When** 올바른 이메일/비밀번호를 입력하면, **Then** 인증되어 메인 화면으로 이동한다.
3. **Given** 사용자가 잘못된 비밀번호를 입력할 때, **When** 로그인을 시도하면, **Then** "이메일 또는 비밀번호가 올바르지 않습니다" 오류 메시지가 표시된다.
4. **Given** 이미 사용 중인 이메일로 회원가입을 시도할 때, **When** 가입을 완료하려 하면, **Then** "이미 사용 중인 이메일입니다" 오류 메시지가 표시된다.

---

### User Story 2 - 보호된 페이지 접근 제어 (Priority: P2)

로그인하지 않은 사용자가 메인 화면, 캘린더, 통계 등 인증이 필요한 페이지에 접근하려 할 때 자동으로 로그인 화면으로 이동한다.

**Why this priority**: 데이터 보안의 기본 요건이다. 이 스토리가 없으면 타인의 할 일 데이터에 접근할 수 있는 보안 취약점이 생긴다.

**Independent Test**: 로그인 없이 `/main` URL 직접 접근 → 로그인 화면으로 리다이렉트 되는지 확인으로 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 비로그인 사용자가 브라우저에서 `/main` URL에 직접 접근할 때, **When** 페이지가 로드되면, **Then** 로그인 화면(`/`)으로 자동 이동한다.
2. **Given** 비로그인 사용자가 `/calendar` 또는 `/stats`에 접근할 때, **When** 페이지가 로드되면, **Then** 로그인 화면으로 자동 이동한다.
3. **Given** 로그인한 사용자가 로그인 화면(`/`)에 접근할 때, **When** 페이지가 로드되면, **Then** 이미 인증됐으므로 메인 화면으로 자동 이동한다.

---

### User Story 3 - 사용자별 데이터 격리 (Priority: P3)

각 사용자는 자신이 만든 할 일만 볼 수 있으며, 다른 사용자의 할 일은 접근할 수 없다.

**Why this priority**: 다수 사용자가 동일 앱을 사용하는 핵심 요건. 수업 후 친구에게 링크를 공유하는 목표 달성에 직접적으로 필요하다.

**Independent Test**: 사용자 A와 B가 각각 로그인하여 할 일을 추가한 뒤, 서로의 목록에 상대방의 할 일이 보이지 않음을 확인으로 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 사용자 A가 할 일 "운동하기"를 추가한 후, **When** 사용자 B가 로그인하여 목록을 확인하면, **Then** "운동하기"가 보이지 않는다.
2. **Given** 로그인한 사용자가 할 일을 추가/수정/삭제할 때, **When** 변경 사항이 저장되면, **Then** 해당 사용자가 다른 기기에서 로그인해도 동일한 데이터가 보인다.

---

### User Story 4 - Google 소셜 로그인 (Priority: P4)

사용자가 Google 계정으로 가입 및 로그인할 수 있다.

**Why this priority**: 이메일/비밀번호 인증(P1)이 완성된 후 추가되는 편의 기능. 별도의 비밀번호 관리 없이 기존 Google 계정을 재사용할 수 있어 진입 장벽을 낮춘다.

**Independent Test**: "Google로 로그인" 버튼 클릭 → Google OAuth 화면 → 인증 완료 → 메인 화면 이동 흐름으로 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 사용자가 로그인 화면에서 "Google로 로그인" 버튼을 클릭할 때, **When** Google 계정 선택 및 권한 동의를 완료하면, **Then** 메인 화면으로 이동하고 로그인 상태가 유지된다.
2. **Given** Google 로그인으로 처음 가입한 사용자가, **When** 다음 방문 시 동일 Google 계정으로 로그인하면, **Then** 이전에 저장한 할 일 목록이 그대로 보인다.

---

### User Story 5 - 로그아웃 (Priority: P5)

로그인한 사용자가 로그아웃할 수 있으며, 이후 인증이 필요한 페이지에 접근하면 로그인 화면으로 이동한다.

**Why this priority**: 기본적인 세션 관리 요건. 공용 컴퓨터 사용 시 다른 사람이 내 계정에 접근하지 못하도록 한다.

**Independent Test**: 로그아웃 버튼 클릭 → 로그인 화면으로 이동 → 브라우저 뒤로가기로 메인 접근 시도 → 로그인 화면으로 재이동 확인으로 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 앱을 사용 중일 때, **When** 로그아웃 버튼을 누르면, **Then** 세션이 종료되고 로그인 화면으로 이동한다.
2. **Given** 로그아웃한 사용자가 브라우저 뒤로가기로 메인 화면에 접근하려 할 때, **When** 페이지가 로드되면, **Then** 로그인 화면으로 리다이렉트된다.

---

### Edge Cases

- 비밀번호를 6자 미만으로 입력하면 "비밀번호는 최소 8자 이상이어야 합니다" 메시지가 표시된다.
- 잘못된 형식의 이메일 입력 시 즉시 유효성 검사 오류가 표시된다.
- Google 로그인 도중 팝업을 닫거나 권한을 거부하면 로그인 화면에 남아있고 오류 메시지가 표시된다.
- 네트워크 오류로 로그인에 실패할 경우 사용자에게 재시도 안내 메시지가 표시된다.
- 세션이 만료된 상태에서 앱을 재방문하면 자동으로 로그인 화면으로 이동한다.
- 이미 로그인된 상태에서 로그인 화면을 직접 방문하면 메인 화면으로 자동 이동한다.

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 이메일과 비밀번호(8자 이상)를 통한 회원가입을 지원해야 한다.
- **FR-002**: 시스템은 이메일과 비밀번호로 로그인을 지원해야 한다.
- **FR-003**: 시스템은 Google OAuth를 통한 소셜 로그인을 지원해야 한다.
- **FR-004**: 시스템은 인증되지 않은 사용자가 보호된 페이지(`/main`, `/calendar`, `/stats`)에 접근할 때 로그인 화면으로 리다이렉트해야 한다.
- **FR-005**: 시스템은 이미 로그인된 사용자가 로그인 화면에 접근할 때 메인 화면으로 리다이렉트해야 한다.
- **FR-006**: 각 사용자는 자신이 생성한 할 일 데이터만 조회·수정·삭제할 수 있어야 한다.
- **FR-007**: 시스템은 로그아웃 기능을 제공해야 하며, 로그아웃 후 세션이 즉시 무효화되어야 한다.
- **FR-008**: 할 일 데이터는 클라우드에 저장되어 로그인한 기기와 무관하게 동기화되어야 한다.
- **FR-009**: 로그인 및 회원가입 오류(잘못된 자격증명, 중복 이메일 등)는 사용자가 이해할 수 있는 한국어 메시지로 표시되어야 한다.
- **FR-010**: 인증 관련 입력 폼(이메일, 비밀번호)은 제출 전 클라이언트 측 유효성 검사를 수행해야 한다.

### Key Entities

- **사용자(User)**: 이메일, 인증 방식(이메일/Google), 가입일 등의 속성을 가지며 할 일 목록의 소유자.
- **세션(Session)**: 로그인 상태를 나타내며, 브라우저를 닫아도 일정 기간 유지되는 인증 토큰.
- **할 일(Task)**: 기존 데이터 모델에 소유자(사용자) 정보가 추가되어 사용자별 데이터 격리를 보장.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 회원가입 완료까지 걸리는 시간이 2분 이내.
- **SC-002**: 로그인 후 메인 화면 진입까지 3초 이내.
- **SC-003**: 보호된 페이지에 비로그인 상태로 접근 시 100% 로그인 화면으로 리다이렉트.
- **SC-004**: 사용자 A의 할 일이 사용자 B의 화면에 노출되는 데이터 유출 케이스 0건.
- **SC-005**: 인증 추가 후 기존 할 일 CRUD 기능(추가, 수정, 삭제, 완료 토글)이 100% 정상 동작.
- **SC-006**: 로그아웃 후 브라우저 히스토리 탐색으로 보호 페이지 접근 시 100% 차단.

## Assumptions

- 사용자는 현대 브라우저(Chrome, Safari, Firefox 최신 버전)를 사용한다.
- Google 소셜 로그인은 Supabase Auth의 Google OAuth Provider 기능을 활용한다.
- 비밀번호 재설정(이메일 찾기) 기능은 이번 스코프에 포함되지 않는다.
- 기존 localStorage 기반 할 일 데이터는 마이그레이션 없이 신규 가입 사용자부터 클라우드 저장이 적용된다.
- 인증 미들웨어는 서버사이드(Next.js Middleware)에서 처리하여 클라이언트 우회를 방지한다.
- 세션은 브라우저를 닫아도 7일간 유지되는 "기억하기" 방식을 기본으로 한다.
- 소셜 로그인(Google)은 Supabase 대시보드에서 OAuth 앱 설정이 완료되어 있다고 가정한다.
